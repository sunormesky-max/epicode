# Epicode v1.1.0 开发文档 — 编程外脑三特性

## 核心边界定义（绝对不可修改）

以下组件构成Epicode的核心物理引擎，任何新特性只能**调用**它们，不能**修改**它们：

| 组件 | 文件 | 禁止操作 |
|------|------|----------|
| Space（空间域） | `domain/space.rs` | 不改四面体/顶点/聚类逻辑 |
| Energy（能量中心） | `engine/energy.rs` | 不改守恒定律/充放机制 |
| Dream（梦境引擎） | `engine/dream.rs` | 不改巩固/合并/归档流程 |
| Pulse（脉冲引擎） | `engine/pulse.rs` | 不改传播/感知机制 |
| Knowledge（知识图谱） | `engine/knowledge.rs` | 不改关系/概念核心逻辑 |
| Storage（持久层） | `engine/storage.rs` | 不改SQLite schema核心结构（只允许ADD COLUMN） |
| Crypto（加密） | `engine/crypto.rs` | 不改加密算法 |
| Cylinder（中央调度器） | `domain/cylinder.rs` | 不改Port/层/锚点机制 |
| Vector（向量层） | `engine/vector.rs` | 不改ONNX推理/余弦相似度 |
| Constitution（宪法） | `engine/constitution.rs` | 不改核心规则 |
| Search Engine | `engine/search_engine.rs` | 不改评分公式核心（刚优化完） |

**允许的操作**：
- 在 `mcp.rs` 中新增MCP工具端点
- 在 `gateway.rs` 中新增公共方法（不改已有方法）
- 在 `scheduler.rs` 中新增调度逻辑（不改已有tick）
- 新增 `.rs` 文件（不拆分现有文件）
- 在 `MemoryPayload` 中新增字段（必须 `serde(default)` + DB migration ADD COLUMN）
- 在 `cloud.rs` 中新增HTTP路由

## 特性一：ctx_load 自动注入

### 1.1 问题
agent必须主动调用memory_search才能获取记忆。如果agent没想到去搜，记忆就浪费了。

### 1.2 设计
新增MCP工具 `ctx_load` — 返回当前用户的高importance记忆摘要，供agent在会话开始时自动注入。

### 1.3 实现方案

**新增文件**：无（在 `mcp.rs` 和 `gateway.rs` 中扩展）

**gateway.rs 新增方法**：
```rust
pub fn load_context(&self, limit: usize) -> Vec<(TetraId, f64, String, Vec<String>)>
```
- 逻辑：获取所有tetra，按 `importance * mass` 降序排列，取top-N
- 返回：(id, importance, content前200字, labels)
- 不改search_engine.rs，纯gateway层读取

**mcp.rs 新增工具**：
```
name: "ctx_load"
params: { project?: string, limit?: number (default 10) }
返回: { memories: [{id, importance, preview, labels}], project_hint?: string }
```

**算法**：
1. `space.all_tetrahedrons()` 获取全部记忆
2. 过滤掉 noise（importance < 0.3 且 access_count = 0）
3. 按 `importance * (1 + mass * 0.1)` 降序排序
4. 取 top `limit`（默认10）
5. 每条返回 content 前200字 + labels + importance

**安全**：
- 只读操作，不消耗能量
- 遵循 SecurityGuard 的 query 验证
- 返回内容长度有限（200字/条），不会撑爆上下文

### 1.4 影响范围
- `gateway.rs`：新增1个方法（不改已有）
- `mcp.rs`：新增1个工具分支（不改已有）
- 无DB变更，无schema变更

## 特性二：Pattern 强制约束

### 2.1 问题
pattern_learn存了编码规范，但agent可能忽略。学到的模式需要变成不可违反的规则。

### 2.2 设计
利用已有的 `constitution.rs` 机制。pattern_recall返回结果时，高置信度的pattern标记为 `enforced: true`。

### 2.3 实现方案

**MemoryPayload 扩展**：
```rust
// 新增字段（serde(default) + DB migration）
pub enforced: bool,  // default: false
```

**gateway.rs 新增方法**：
```rust
pub fn get_enforced_patterns(&self) -> Vec<(TetraId, String)>
```
- 逻辑：搜索所有 labels 包含 "pattern" 且 enforced=true 的记忆
- 返回pattern内容列表

**mcp.rs 扩展**：
1. `pattern_learn` 新增参数 `enforced: bool (default false)`
   - 如果 enforced=true，记忆的 `enforced` 字段设为 true
2. `pattern_recall` 返回值新增 `enforced: bool` 标记
3. 新增工具 `enforced_rules`：
   ```
   name: "enforced_rules"
   params: { project?: string }
   返回: { rules: [{id, pattern, project, enforced}], count: number }
   ```
   返回所有 enforced=true 的规则，供agent作为system prompt的硬约束注入

**DB变更**：
```sql
ALTER TABLE tetrahedrons ADD COLUMN enforced INTEGER NOT NULL DEFAULT 0;
```

### 2.4 影响范围
- `tetra.rs`：MemoryPayload新增1个bool字段（serde(default)）
- `storage.rs`：3处INSERT语句加字段 + 1处migration + 1处load加字段
- `gateway.rs`：新增1个方法
- `mcp.rs`：扩展pattern_learn + pattern_recall + 新增1个工具
- 不改search_engine, dream, energy, pulse, knowledge

### 2.5 enforced pattern 的生命周期
- 创建：pattern_learn(enforced=true)
- 搜索：pattern_recall 返回 enforced 标记
- 强制：enforced_rules 返回所有硬约束
- 撤销：memory_update(enforced=false) 可降级
- 梦境：dream.rs不evict enforced=true的记忆（即使importance低）

## 特性三：项目级记忆分组

### 3.1 问题
不同项目的记忆混在一起。Rust项目的经验和Python项目的经验应该隔离。

### 3.2 设计
利用已有的 **label机制**。不需要新字段，只需要在MCP层面强制要求project参数，并在搜索/列表时自动过滤。

### 3.3 实现方案

**零新字段、零DB变更**。纯MCP层逻辑。

**mcp.rs 改动**：

1. `memory_create` — 如果 labels 中没有 `project:` 前缀的标签，自动注入当前project名
   - 检测方式：遍历 labels，如果没有 `project:xxx` 标签，自动添加 `project:{current_project}`
   - current_project 来自 `ctx_load` 或 `session_summary` 中记录的最近项目

2. `memory_search` — 新增可选参数 `project?: string`
   - 如果提供 project，在搜索结果后过滤：只返回 labels 含 `project:{project}` 的结果
   - 如果不提供，返回所有（保持向后兼容）

3. `memory_list` — 同上，支持 project 过滤

4. `memory_recall` — 同上

5. 新增工具 `project_list`：
   ```
   name: "project_list"
   params: {}
   返回: { projects: [{name, memory_count, last_active}], total: number }
   ```
   - 逻辑：扫描 label_index 中所有 `project:` 前缀的标签，统计数量

6. 新增工具 `project_switch`：
   ```
   name: "project_switch"
   params: { project: string }
   返回: { project, enforced_rules: [...], recent_memories: [...] }
   ```
   - 逻辑：切换当前项目上下文，返回该项目的强制规则+最近记忆

### 3.4 影响范围
- `mcp.rs`：扩展4个现有工具 + 新增2个工具
- `gateway.rs`：新增 `list_projects()` 和 `project_context()` 方法
- 零DB变更，零字段变更
- 不改search_engine, dream, energy, pulse, knowledge, storage

## 实施顺序

```
Phase 1: ctx_load（最简单，纯读取，零风险）
  └─ gateway.rs: load_context()
  └─ mcp.rs: ctx_load tool
  └─ 测试验证

Phase 2: enforced patterns（需要DB migration，中风险）
  └─ tetra.rs: enforced字段
  └─ storage.rs: migration + INSERT/LOAD
  └─ gateway.rs: get_enforced_patterns()
  └─ mcp.rs: 扩展pattern_learn/recall + enforced_rules
  └─ dream.rs: enforced记忆不evict
  └─ 测试验证

Phase 3: 项目分组（纯label逻辑，零DB，低风险）
  └─ gateway.rs: list_projects() + project_context()
  └─ mcp.rs: 扩展create/search/list + project_list/switch
  └─ 测试验证

Phase 4: 集成测试 + 部署
  └─ cargo check
  └─ 服务器编译部署
  └─ MCP 22项 + 新工具测试
```

## 风险评估

| 风险 | 等级 | 缓解 |
|------|------|------|
| DB migration失败 | 低 | ADD COLUMN + DEFAULT，与importance同模式 |
| enforced字段被滥用导致记忆不清理 | 低 | dream.rs改为"不evict"但仍可merge |
| project标签导致搜索召回不足 | 低 | project过滤是可选的，不影响无参搜索 |
| ctx_load返回内容过长 | 低 | 硬限200字/条 × 10条 = 2000字 |
| 影响核心物理引擎 | **零** | 所有改动在gateway/mcp层，不触碰核心 |
